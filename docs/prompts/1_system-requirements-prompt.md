# システム要件抽出プロンプト（Cursor用｜西川農園コーポレートサイト）

あなたは **プロダクトマネージャー（PdM）とシステムアーキテクトの両方の実務経験を持つエキスパート**として、以下の入力情報をもとに、西川農園のコーポレートサイトに最適な **システム要件を統合・分析し、簡潔かつ実装可能な形で導き出してください。**

本プロジェクトは「売上を直接生まない名刺代わりのコーポレートサイト」であり、**スピード・シンプルさ・運用しやすさ**を最優先とします。

---

## 入力情報

### ビジネスモデル・プロダクト・制約条件情報

`/docs/input` フォルダ内のファイルを参照してください。  
これらのファイルには、以下のような情報が含まれている可能性があります。

- プロジェクト背景・目的  
- 農園の特徴（地域性、学生参画、取り組み内容）  
- 想定ユーザー（取引先、一般消費者、学生）  
- MVPの範囲（固定ページ中心、問い合わせフォームのみ）  
- 技術スタック前提（Next.js / Supabase / Vercel / Cursor 等）  
- 制約条件（低予算、自分たちで開発、運用負荷を極力下げたい）

**全ての利用可能な情報を統合し、「過不足のない」システム要件を定義してください。**

---

## 不足情報への対応方針

入力ファイルが存在しない、または情報が不足している場合は、以下のルールで対応してください。

### 1. 不足情報の明確化
- 不足している情報を具体的に列挙する  
- 以下のカテゴリごとに整理する  
  - ビジネス要件  
  - ユーザー要件  
  - 技術・運用要件  

### 2. 情報収集の質問提示
- 不足情報を補うための質問を提示する  
- 各質問に **重要度（高・中・低）** を付ける  
- コーポレートサイトとして不要に重い質問は避ける  

### 3. 仮定の設定
- 情報が不足していても作業が止まらないよう、合理的な仮定を置く  
- 仮定は必ず明示する  
- 「学生が運用」「初期ユーザーは少数」など現実的な前提を優先する  

### 4. 情報不足による影響整理
- 要件定義や実装に与える影響を簡潔に説明する  
- 影響度が高い不足情報のみ、優先的な追加取得を促す  

---

## 出力形式

分析結果は `/docs/output/system_requirements.md` に **Markdown形式**で出力してください。  
以下の構成・粒度を必ず守ってください。

```markdown
# システム要件定義書

## システム概要
[300〜500文字程度で、コーポレートサイトの目的・位置づけ・利用シーンを簡潔にまとめる]

1. システムの目的
   - 解決する課題（例：公式情報が散在している）
   - 目指す状態（例：安心してURLを共有できる）
   - 期待される効果（信頼性向上・関係構築）

2. 主な機能
   - コア機能（固定ページ中心、3〜5個）
   - 差別化要素（学生参画・ストーリー性など、2〜3個）
   - 最低限の動的機能（問い合わせ等）

3. 提供する価値
   - 閲覧者（取引先・一般ユーザー）への価値
   - 農園・プロジェクト側への価値
   - 地域・社会的価値（該当する場合）

4. 実現方法の概要
   - 技術的方針（静的生成中心、SaaS活用）
   - 外部連携ポイント（フォーム、DB）
   - 将来拡張を阻害しない設計方針

## 情報の前提と仮定
- 使用した入力ファイル
- 不足情報に対して置いた仮定

## 機能要件
- 各ページ・機能の役割と簡潔な仕様
- MVP外の機能は明確に区別すること

## 非機能要件
- パフォーマンス（想定ユーザー規模を明示）
- セキュリティ（最低限）
- 運用性（学生でも回せるか）
※ 各要件について「なぜそう設定したか」の根拠を簡潔に書くこと

## 技術スタック
以下を第一候補として検討し、妥当性を評価してください。

- フロントエンド: Next.js (React)
- 言語: TypeScript
- バックエンド: Next.js App Router
- データベース: Supabase (PostgreSQL)
- 認証: 原則不要（管理画面なし前提）
- ホスティング: Vercel

※ 選定理由は「開発効率・学習コスト・運用負荷・コスト」を重視して説明すること

## 開発ロードマップ
- フェーズ0：最低限公開できる状態
- フェーズ1：改善・微調整
※ フェーズ分けの理由を簡潔に記載すること

## リスク分析
以下のカテゴリで整理すること。
- 技術的リスク
- 運用リスク
- 情報発信上のリスク

## UI/UXの方向性
- デザインコンセプト（例：温かいが洗練されている）
- トーン＆マナー（若者向け・農業へのリスペクト）
- デバイス方針（モバイルファースト）
- 参考サイト（2〜3件）
