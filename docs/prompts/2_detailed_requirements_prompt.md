# 詳細要件定義書作成プロンプト（Cursor用｜西川農園コーポレートサイト）

あなたは **経験豊富なプロダクトマネージャー兼フルスタックエンジニア**です。  
以下の指示に従い、西川農園のコーポレートサイトについて、**実装に直結するレベルの詳細要件定義書**を作成してください。

本プロジェクトは以下の性質を持ちます。

- 売上を直接生まない「名刺代わり」のコーポレートサイト
- 静的ページ中心、動的機能は最小限（問い合わせフォームのみ）
- 学生主体で開発・運用
- バイブコーディング（Cursor前提）
- 過剰設計を避け、シンプルかつ拡張可能な構成を重視

---

## 入力情報

1. **システム要件定義書**  
   `docs/output/system_requirements.md`  
   - 本ファイルを最重要インプットとする  
   - システム概要、機能要件、非機能要件、UI/UX方針を正確に反映すること  

2. **要件定義書テンプレート**  
   `docs/template/Requirements_Specification_Template.md`  
   - 出力ドキュメントは **構造・見出し・順序を厳密に遵守**すること  

3. **補足情報（必要に応じて）**  
   `docs/input/` フォルダ  
   - system_requirements.md だけでは埋まらない項目がある場合のみ参照  

---

## 出力形式

- **出力ファイル**: `docs/output/detailed_requirements_specification.md`
- **形式**: テンプレートに完全準拠した Markdown

---

## 実行手順

1. `system_requirements.md` と `Requirements_Specification_Template.md` を完全に理解する  
2. system_requirements.md の内容をテンプレート各章へ正確にマッピングする  
3. テンプレートの空欄・抽象部分を、以下の指示に従って具体化する  
4. 情報不足がある場合は、合理的な仮定を置き **(仮定)** と明記する  

---

## 詳細指示（重要）

### 1. プロジェクト概要
- `system_requirements.md` の「システム概要」をベースに記述
- コーポレートサイトとしての位置づけを明確にする

---

### 2. ビジネス要件
- リーンキャンバスは **簡易要約レベル**でよい
- KPI/KGIは定量でなく **定性的指標中心**で設定
- 非収益プロダクトであることを明記する

---

### 3. ユーザー要件
- ペルソナは **1〜2名で十分**
  - 取引先（企業・卸）
  - 一般閲覧者（20〜40代）
- ユーザーストーリーは **3〜5個**
- MVP定義は「最低限外部に共有できる状態」を基準にする

---

### 4. 機能要件
- MoSCoW分類で機能一覧を作成
- 主要機能は以下を想定し、**3機能程度のみ詳細化**
  - トップページ
  - 農園紹介／ストーリー
  - お問い合わせフォーム
- 各機能について以下を簡潔に記述
  - 概要
  - ユースケース
  - 正常系フロー
  - 例外系（最小限）
  - UI要件

※ 管理画面・認証・権限管理は原則対象外

---

### 5. UI/UX設計
- `system_requirements.md` のUI/UX方針を具体化
- 以下を必ず含める

1. デザインコンセプト  
   - 温かい／土／人の顔が見える  
   - 若者向けでかっこいい  
   - 農業へのリスペクト  

2. デザイン要素（提案）
   - カラーパレット（自然色ベース）
   - タイポグラフィ（可読性重視）

3. 画面一覧
   - トップ
   - 農園紹介
   - プロジェクト紹介
   - 作物紹介
   - メンバー
   - お問い合わせ

4. 画面遷移図（Mermaid形式）

5. ワイヤーフレーム（テキストベース）
   - トップページ
   - 農園紹介ページ
   - お問い合わせページ

---

### 6. 非機能要件
- system_requirements.md を整理・転記
- 以下を重視して記述
  - パフォーマンス（初期ユーザー少数想定）
  - セキュリティ（最低限）
  - 運用性（学生でも維持できるか）

---

### 7. データベース設計
- お問い合わせ機能を前提に **最小限**
- 以下を生成すること

1. ER図（Mermaid形式）
2. テーブル定義
   - inquiries テーブル程度で十分
   - カラム名、型、制約を明記

---

### 8. インテグレーション要件
- 想定外部連携
  - Supabase
  - Vercel
- REST形式でAPI仕様を定義
  - POST /api/contact など
- 認証APIは不要

---

### 9. 技術選定とアーキテクチャ
以下を前提として記述すること。

- フロントエンド: Next.js (App Router)
- 言語: TypeScript
- データベース: Supabase (PostgreSQL)
- ホスティング: Vercel

必須アウトプット：
1. アーキテクチャ概要図（Mermaid）
2. コンポーネント階層図（Mermaid）
   - Server Components / Client Components を明示
3. 主要コンポーネント3つ程度について
   - Props（型）
   - Server / Client区分
   - 状態管理方針

---

### 10. リスクと課題
- system_requirements.md をベースに整理
- 以下の観点で記述
  - 技術
  - 運用
  - 情報発信

---

### 11. ランニング費用と運用方針
- 無料枠前提で概算
- 運用体制は「学生が無理なく回せる」前提
- (仮定) を明記すること

---

### 12. 変更管理
- GitHub Issue / PR 前提
- 要件変更は軽量に扱う

---

### 13. 参考資料
- system_requirements.md
- デザイン参考サイト
- 使用SaaS公式ドキュメント

## 評価基準（最重要）

- テンプレート構造への完全準拠
- 実装時に「次何作る？」で迷わない具体度
- 過剰設計になっていないこと
- Cursorでそのままコードに落とせる内容であること
- 仮定が明示されていること
